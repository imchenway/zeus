# TASK_20260617_001 Zeus 项目图谱生成与抽屉布局返修

## 1. 用户反馈

用户反馈两类问题：

1. 项目代码的图谱无法生成。
2. 点击打开图谱后，浮层抽屉内部内容仍然乱七八糟。

本次先按缺陷排查，不直接改源码；结论用于后续 develop 阶段一次性修复。

## 2. 现象与影响

### 2.1 图谱无法生成

现有真实扫描命令 `pnpm test:real-scan` 可以成功生成当前 Zeus 仓库图谱，结果为：

- `fileCount=174`
- `symbolCount=17036`
- `nodeCount=17036`
- `edgeCount=32848`
- `viewCount=7`

因此扫描引擎本体不是主要故障点。用户侧“项目代码图谱无法生成”的更高置信根因在桌面端项目页触发链路：项目页按钮调用的是全局当前仓库扫描与全局图谱读取，而不是所选项目的项目级 API。

### 2.2 抽屉内容混乱

当前 `CodeMapView` 在一个抽屉内纵向平铺了：

- Spatial Graph Studio 顶部说明
- 视图切换
- 性能条
- 搜索筛选
- 图谱问答
- 问答历史
- WebGL/Sigma 运行时
- SVG/React Flow 局部图
- Mermaid 预览与导出
- 节点显示控制
- 节点列表
- 边列表
- 边详情
- 节点详情

这违反“一个页面只做一件核心事情”和“低频功能收纳到下一层”的前端约束。抽屉本应服务“查看当前项目图谱”，但现在像把 Code Map 功能清单塞进抽屉。

同时，`styles.css` 存在全局规则：`.macos-ai-app p, .macos-ai-app em, .macos-ai-app small { display: none !important; }`，会隐藏图谱抽屉内大量来源、说明、状态、性能、导出反馈文本，进一步造成内容层级断裂。

## 3. 根因

### 3.1 根因 A：项目级图谱 API 没有接入桌面端项目页

当前已有 API client 能力：

- `scanProject(projectId)` -> `POST /api/projects/:projectId/scan`
- `loadProjectGraphView(projectId, viewType)` -> `GET /api/projects/:projectId/graph/views/:viewId`
- `searchProjectGraph(projectId, input)` -> `GET /api/projects/:projectId/graph/search`

但 `App.tsx` 的项目页仍只暴露：

- `onScanCurrentGraph`
- `onLoadGraphView`
- `onSearchGraph`

项目页按钮实际调用：

- “扫描项目” -> `scanCurrentGraph()` -> `onScanCurrentGraph()` -> `/api/graph/scan-current`
- “打开图谱” -> `handleCodeMapAction()` -> `openGraphView()` 或 `scanCurrentGraph()` -> 全局当前仓库图谱

这意味着用户选中某个真实项目后，界面并没有按该项目 ID 生成和读取图谱。

### 3.2 根因 B：图谱上下文多处绑定第一个项目而非当前选中项目

`App.tsx` 中多个图谱/任务函数依赖 `firstProjectId`：

- `askGraph`
- `loadGraphConversations`
- `loadGraphConversationDetail`
- `archiveGraphConversation`
- `restoreGraphConversation`
- `createTaskFromGraphNode`
- `createTaskFromGraphConversation`

当用户在项目列表选择第二个或后续项目时，这些操作仍可能绑定列表第一个项目，导致“当前项目”和“图谱上下文”错位。

### 3.3 根因 C：抽屉内部没有重新做信息架构，只是把旧 Code Map 页面塞进浮层

`WorkspaceDrawer` 已经是固定定位浮层，但 `CodeMapView` 内部仍是旧式功能平铺。浮层解决了“不要在当前页展开”的问题，没有解决“打开后内容必须按操作路径递进”的问题。

### 3.4 根因 D：全局隐藏文本标签导致来源与说明消失

`styles.css` 的全局隐藏规则为了压缩首层文案，把 `p/em/small` 全部隐藏；但 Code Map 抽屉需要这些元素呈现来源、状态、说明和错误恢复。该规则不应作为全局规则存在，应改为针对首层冗余文案的精确收纳或专用类。

## 5. 契约变更

### 5.1 前端 props 契约

建议新增或调整：

- `onScanProjectGraph(projectId: string): Promise<DashboardSnapshot>`
- `onLoadProjectGraphView(projectId: string, viewType?: GraphViewType): Promise<GraphViewSnapshot>`
- `onSearchProjectGraph(projectId: string, query: string, nodeType?: string, edgeType?: string, minConfidence?: number): Promise<GraphSearchResult>`

保留全局 `onScanCurrentGraph/onLoadGraphView/onSearchGraph` 作为当前 Zeus 自检或兼容入口，但项目页优先使用项目级契约。

### 5.2 UI 契约

图谱抽屉只把“图谱舞台”作为首层：

- 首层必须展示：项目名、扫描状态、节点/边/视图数量、视图切换、图谱画布、选中节点摘要。
- 首层不再直接平铺：问答历史、Mermaid 导出源码、完整节点列表、完整边列表、二跳影响详情。
- 低频能力放入抽屉内二级区域：`inspect`、`ask`、`export`、`history`。
- 来源、错误、空态、性能、导出反馈不能被全局 CSS 隐藏。

### 5.3 后端/数据库契约

本次不需要改后端接口、SQLite schema 或新增依赖。项目级扫描与项目级视图 API 已存在，修复重点是 renderer 接入与 UI 结构。

## 6. 推荐修法

### 6.1 图谱生成链路

1. 在 `App` props 中接入项目级图谱回调。
2. 增加 `activeProjectId = selectedProject?.id ?? firstProjectId`，所有项目页图谱操作使用当前选中项目。
3. “扫描项目”优先调用 `onScanProjectGraph(activeProjectId)`，成功后调用 `onLoadProjectGraphView(activeProjectId, 'architecture')`。
4. “打开图谱”若已有视图则调用项目级 view；若无视图则触发项目级扫描。
5. 图谱搜索、问答、从节点创建任务、问答创建任务都绑定 `activeProjectId`。
6. 捕获错误时写入 `localError`，不要只把 `scanState` 置为 failed，避免用户看不到失败原因。

### 6.2 抽屉信息架构

采用“主舞台 + 右侧检查器 + 二级工具区”而不是继续纵向平铺：

- 顶部：项目/视图状态条，含扫描状态、节点数、边数、重新扫描。
- 主区：图谱画布优先，节点/边详情在右侧检查器。
- 工具区：搜索筛选作为紧凑 toolbar；问答、历史、Mermaid 导出收纳到 segmented tabs 或 `details`。
- 空态：没有 view 时只展示为什么为空和一个主按钮，不再显示大量空模块。

### 6.3 样式修复

- 删除全局 `.macos-ai-app p/em/small { display:none !important; }`。
- 对确实需要隐藏的首层冗余说明改为专用类，例如 `.first-layer-copy-muted` 或结构上不渲染。
- 为 Code Map 抽屉补充局部布局类，确保长路径、长节点名、长 AI 回答、Mermaid 源码都可滚动/换行，不撑爆抽屉。

## 7. 方案对比

| 方案 | 内容 | 优点 | 缺点 | 结论 |
|---|---|---|---|---|
| A 最小修复 | 仅把按钮改成项目级 API，删除全局隐藏文本 | 改动小，能解决“无法生成”和明显断裂 | 抽屉仍是功能平铺，用户“乱七八糟”反馈不会彻底解决 | 不推荐 |
| B 推荐修复 | 项目级 API 接入 + 当前项目绑定 + Code Map 抽屉信息架构重组 + CSS 精确化 | 同时修复功能错链路和体验根因，不新增依赖 | 改动涉及 App.tsx/CSS/测试，需谨慎 TDD | 推荐 |
| C 重做 Code Map 独立页面 | 抽屉只入口，图谱回到独立一级/全屏页 | 图谱空间最大 | 违背用户要求的收纳式壳层，不符合已确认架构 | 不推荐 |
