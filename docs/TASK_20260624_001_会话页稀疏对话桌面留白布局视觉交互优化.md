# TASK_20260624_001 会话页稀疏对话桌面留白布局视觉交互优化

## 1. 背景与目标

用户要求基于截图找出任意页面的布局、视觉、交互问题并优化。本轮选择当前真实 Electron 窗口中可见的「项目会话」页。

截图现象：默认 1240×820 窗口下，当前会话只有「用户要求」和「等待下一步」两条内容时，消息流贴近顶部，composer 固定在底部，中间出现大块无意义留白。用户视线需要跨越大片空白才能从上下文移动到输入区，当前线程和下一步操作割裂。

唯一核心目标：稀疏会话场景下，首屏仍应把「当前消息上下文 + composer」作为一个连续工作区，方便用户立刻补充要求或继续发送到 CLI。

## 2. 当前问题

| 编号 | 现象 | 影响 | 根因判断 |
|---|---|---|---|
| P1 | 消息流停在详情区域顶部，composer 固定在底部 | 当前上下文与输入动作断开，用户需要跨越大片空白 | 会话线程使用 `auto minmax(0, 1fr) auto`，稀疏内容没有根据 composer 重新聚拢 |
| P2 | 中间留白没有表达任何状态 | 看起来像布局没加载完或内容丢失 | 空白不是有意的等待态，也没有操作意义 |
| P3 | 若直接压缩全部会话消息区，会影响长事件流 | 长对话需要保留顶部阅读和滚动空间 | 必须区分无事件稀疏会话与已有事件的活跃会话 |

## 3. 页面核心目标判断

1. 唯一核心目标：选择或查看当前会话，并在底部 composer 继续推进任务。
2. 用户当前最需要做什么：读懂当前任务要求与等待状态，然后补充要求或继续发送。
3. 首层必须展示：当前线程标题、最新消息/等待状态、composer 输入与发送动作。
4. 应收纳到下一层：运行环境、上下文、代码变更、模板、完整 Runtime 日志。
5. 禁止回退：不能把会话页做成后台详情面板、卡片堆叠或大面积空白舞台。

## 4. 受影响目录

| 路径 | 影响 |
|---|---|
| `apps/desktop/src/renderer/App.tsx` | 为会话线程根节点增加 `data-conversation-density="sparse|active"`，用于区分稀疏会话和已有事件的活跃会话。 |
| `apps/desktop/src/renderer/styles.css` | 新增桌面宽度下的稀疏会话布局覆盖，让消息流靠近 composer，减少无意义中段空白。 |
| `apps/desktop/test/app-shell-layout.test.tsx` | 新增 TDD 回归测试，锁定 sparse/active DOM 契约与 CSS 覆盖。 |
| `docs/TASK_20260624_001_会话页稀疏对话桌面留白布局视觉交互优化.md` | 记录需求、契约、测试矩阵、实施顺序、风险与回滚。 |

## 5. 不受影响边界

| 范围 | 是否修改 | 原因 |
|---|---:|---|
| 后端 / SQLite / IPC | 否 | 本轮只改 renderer DOM 标记和 CSS，不改变任务、事件或会话数据。 |
| Runtime / Git / Telegram 业务逻辑 | 否 | 只调整稀疏会话视觉密度，不改变运行、暂停、取消、发送等行为。 |
| 新对话空态 | 否 | 新对话已有独立 zero state 和 starter suggestion，不复用 sparse 对话规则。 |
| 紧凑宽度布局 | 否 | 860px 以下已有截图驱动覆盖，本轮只处理桌面默认宽度。 |
| 构建依赖 / CI | 否 | 不新增依赖，不改构建配置。 |

## 6. 契约变更

### 6.1 DOM 契约

- 当前会话线程根节点 `.conversation-thread-shell.conversation-codex-surface` 新增 `data-conversation-density`。
- `taskEvents.length === 0` 时为 `sparse`。
- `taskEvents.length > 0` 时为 `active`。
- 该属性只表达布局密度，不改变状态机、不改变可访问名称、不改变业务状态。

### 6.2 CSS 布局契约

- 新增 `@media (min-width: 861px) and (max-width: 1420px)` 桌面窗口覆盖。
- 仅选择 `.conversation-thread-shell[data-conversation-density='sparse']` 下的会话消息与 composer。
- `.conversation-message-stage` 与 `.conversation-message-flow` 在稀疏场景下底部聚拢，避免中段空白。
- `.conversation-codex-composer` 顶部间距收敛到 8px，保持 composer 和消息上下文连续。
- 活跃会话、长事件流、新对话、窄屏布局不受影响。

## 9. 风险与回滚

| 风险 | 影响 | 控制 | 回滚 |
|---|---|---|---|
| 稀疏对话消息下移过多 | 顶部阅读区域显得空 | 只在默认桌面宽度和无事件对话生效，标题仍留在顶部 | 删除 sparse CSS 块和 DOM 属性即可。 |
| 活跃长会话被错误底部对齐 | 长事件流阅读体验下降 | DOM 明确用 `taskEvents.length` 区分 active | 恢复为无 `data-conversation-density` 选择器。 |
| 与 860px 以下紧凑覆盖冲突 | 小屏布局回归 | 使用 `min-width: 861px` 排除已有紧凑覆盖 | 删除本轮 media query。 |
| 视觉截图依赖本地真实数据 | 不同本机任务数量可能略有不同 | 验证聚焦在当前选中稀疏会话的空间关系 | 以 DOM/CSS 测试作为稳定契约，截图作为人工证据。 |

## 12. 实施结果

- `App.tsx`：当前会话线程根节点增加 `data-conversation-density`，以真实事件数区分 `sparse` 与 `active`。
- `styles.css`：新增桌面默认宽度下的稀疏会话覆盖，只把无事件对话的消息区和 composer 聚拢；活跃长对话、紧凑宽度、新对话空态不受影响。
- `app-shell-layout.test.tsx`：新增红绿测试，锁定 sparse/active DOM 契约和 CSS 覆盖，防止后续重构重新制造中段空白。

## 13. 回滚方式

如后续真实使用发现稀疏会话下移过度，可一次性回滚以下三处：

1. 删除 `App.tsx` 中 `.conversation-thread-shell` 的 `data-conversation-density` 属性。
2. 删除 `styles.css` 中带有「稀疏会话桌面留白最终覆盖」注释的 media query。
3. 删除 `app-shell-layout.test.tsx` 中 `sparse desktop conversations` 测试。

该回滚不涉及后端、数据迁移、IPC、Runtime 或打包配置。
