# TASK_20260624_002 任务页去 HERO 化与可用性重构

## 任务目标
- 任务页进入后直接展示任务表格工作台，移除顶部 HERO / `task-detail-status-row` / `zeus-object-toolbar`。
- 表格顶部只保留一行紧凑工具栏：项目与计数 metadata、搜索、状态筛选、排序、标签筛选、新任务。
- 筛选即时作用于当前任务列表，不再提供显式「筛选」按钮。
- 点击整行任务打开详情抽屉；任务描述、事件、Runtime 推进、完成、取消、重试都从首屏移入抽屉。

## 契约变更
- 前端 DOM 契约
  - 任务页 DOM 不再包含 `task-detail-status-row` 与任务页内的 `zeus-object-toolbar`。
  - 任务页 toolbar 使用 `task-table-compact-toolbar`。
  - 任务行带 `data-task-row-action="open-detail"`，整行是打开详情抽屉的唯一行级动作。
- 交互契约
  - 搜索、状态、排序、标签变更即时影响本地当前任务列表。
  - Enter / Space 可打开当前聚焦任务详情；上下/Home/End 可移动任务行焦点。
  - 关闭抽屉后由 `WorkspaceDrawer` 恢复焦点到触发行。
- 数据/API 契约
  - 不改后端、SQLite、IPC、任务 API、Runtime API。
  - 不新增依赖，不引入 mock 数据。

## 2026-06-24 返修：新任务反馈与抽屉产品化

### 用户反馈
- 点击任务页 toolbar 的「新任务」后没有可感知反馈。
- 任务详情抽屉视觉粗糙，信息层级和动作可用性不足。

### 根因
- 「新任务」创建后只刷新 snapshot 与事件，没有清理当前本地筛选、没有选中新任务、没有打开详情抽屉；当排序/筛选或视觉位置不明显时，用户感知为点击无效。
- 抽屉内容按“标题 + metadata + 块 + 三组按钮”直译功能清单，缺少窄幅 sheet 约束、摘要行、事实行、事件行和固定动作轨。

### 修法
- 新增 `createProjectDefaultTask({ openCreatedTaskDrawer })`：记录创建前任务 id，创建后定位真实新增任务。
- 任务页 toolbar 改用 `createTaskFromTaskToolbar`：创建后清空搜索/状态/标签筛选，选中新任务，打开 `WorkspaceDrawer`。
- 会话页现有新建入口仍保留原行为，不强行跳任务抽屉。
- `TaskDetailDrawerContent` 重构为产品化 sheet：摘要行、三列事实、描述区、事件行、底部 sticky action rail。
- CSS 追加 `任务详情抽屉产品化最终覆盖`：任务抽屉收窄到 720px、内容去粗糙块状分组、动作区固定在底部并保持移动端单列。

## 2026-06-24 返修三：任务抽屉右侧 Sheet 与关闭按钮

### 截图问题
- 任务详情抽屉从页面中部横切 toolbar，视觉上像一块临时浮层压住表格，而不是右侧产品 sheet。

- 关闭按钮仍暴露浏览器默认 button 外观，和 Zeus macOS 产品控件语言不一致。

### 根因
- `WorkspaceDrawer` 基础变量默认给所有抽屉预留顶部 56px，适合部分会话/二级抽屉，但任务详情在任务页首屏会切穿 toolbar。

### 修法
- `WorkspaceDrawer` 关闭按钮新增 `workspace-drawer-close-button` class，并保留 `aria-label`。
- 任务抽屉根层新增 `:has(.task-drawer)` 最终覆盖：`--zeus-drawer-backdrop-inset-block: 0 0`、`--zeus-drawer-inset-block: 0 0`，让任务详情从右侧完整贴边进入，不再切 toolbar。
- 任务抽屉最终覆盖写入实际宽度：`inline-size: min(720px, calc(100vw - 236px))`，并同步 `max-inline-size/min-inline-size`。
- 抽屉 chrome 和关闭按钮改为轻量 macOS 产品控件：无默认边框、28px 命中高度、hover/focus 使用 source-list hover 与统一 focus token。

## 2026-06-24 返修五：任务 toolbar 鼠标命中区修复

### 用户反馈
- 任务页顶部搜索框、状态下拉、排序下拉、标签输入和「新任务」按钮的鼠标触发很奇怪。
- 鼠标只有从某个特定角度滑入时才高亮，然后才能点击。

### 根因
- Zeus 使用 hidden titlebar，并在窗口顶部放置 `window-drag-strip` 作为透明拖拽层，高度 56px、`z-index: 30`。
- 任务页去 HERO 化后，toolbar 贴近窗口顶部；控件上半部分落在透明拖拽层覆盖范围内。
- 全局 `button/input/select/label` 虽然有 `-webkit-app-region: no-drag`，但无法穿透一个更高层级、独立接收 pointer 的透明拖拽 div，所以真实可点击区域和视觉控件区域错位。

### 修法
- 为任务页 `task-table-compact-toolbar` 增加最终覆盖：`position: relative; z-index: 31; -webkit-app-region: no-drag; pointer-events: auto;`。
- 为 toolbar 内 `input/select/button` 显式保留 `pointer-events: auto`，确保 hover/click/focus 由可见控件本身接收。
- 不改 `window-drag-strip` 本身，避免破坏窗口顶部空白区域拖拽能力；只把任务 toolbar 作为交互岛升到拖拽层之上。

### 测试与验证
- 红灯：新增 `keeps task toolbar controls above the hidden titlebar drag strip...` 先失败，命中缺少 `任务页 toolbar 命中区最终覆盖`。
- 绿灯：`pnpm vitest run apps/desktop/test/app-task-controls-rendering.test.tsx --testNamePattern "toolbar controls above|命中区" --reporter=verbose` 通过，1 passed / 19 skipped。

- 回归：`pnpm vitest run apps/desktop/test/task-workspace-model.test.ts --reporter=verbose` 通过，3/3。
- 回归：`pnpm vitest run apps/desktop/test/app-shell-layout.test.tsx --testNamePattern "task|任务|table|drawer|hero|button|parking|id|focus|sheet|关闭|task-filter|status-row|toolbar|拖拽|drag" --reporter=verbose` 通过，93 passed / 70 skipped。
- 质量：`pnpm format:check && pnpm lint && pnpm typecheck && git diff --check` 通过。

## 2026-06-24 返修六：跨页面顶部交互命中区扩扫

### 用户反馈
- 任务页 toolbar 的搜索、下拉和「新任务」已暴露出鼠标命中区错位问题。
- 需要继续检查其它页面的抽屉、按钮、下拉等是否存在同类问题；存在则一次性修复。
- 额外约束：不得前台打开或操作 Zeus，避免影响用户正在使用电脑。

### 扩扫结论
- 同类风险不是某个下拉组件单独损坏，而是 hidden titlebar 的透明 `window-drag-strip` 位于窗口顶部 56px、`z-index: 30`，会盖住任何贴近窗口顶部且层级较低的交互面。
- 已确认任务页之外同属风险面的页面级交互区：
  - 代码页：`zeus-object-toolbar`、`project-code-context-rail`、图谱顶部 `graph-view-selector-row` / `graph-search-control-grid`。

## 2026-06-25 设计：个人 × AI 协作任务字段与自定义列

### 设计结论
- 任务页不走团队协作模型，不引入负责人、处理人、@我、待我处理、SLA、逾期等组织流程字段。
- Zeus 任务页定位为“个人向 AI 派活、跟踪 AI 执行、收口结果”的本地工作队列。
- 本轮设计允许修改任务数据契约与本地 SQLite 迁移，用真实任务编码和字段偏好解决“字段不够 AI native、不可自定义、缺少任务编码”的根因。
- 不新增外部依赖，不改 AI Runtime 的执行协议，不引入 mock 数据。

### 用户 / 场景 / 业务目标
- 用户：个人开发者、技术决策者、开源维护者，在本机用 Zeus 管理真实项目任务和 AI CLI 执行闭环。
- 场景：用户进入任务页后，需要快速判断每个任务是什么、来自哪里、AI 是否已启动、当前卡在哪一步、下一步该做什么。
- 目标：任务列表从“普通后台任务表”升级为“个人 × AI 工作队列”，同时允许用户按个人偏好显示字段。
- 成功标准：首屏不回到 HERO / 卡片 / 按钮堆；默认列能体现 AI 协作；任务编码稳定可见；字段可隐藏、排序、恢复默认；详情抽屉同源展示任务编码和下一步。

### 当前现状
- 前端 `TaskRecord` 当前只暴露 `id / projectId / title / description / status / templateId / tags`。
- SQLite `tasks` 表已存在 `priority / created_from / source_context_json / created_at / updated_at`，但前端任务列表没有完整使用。
- `taskWorkspaceModel.ts` 当前排序里 `createdAt / updatedAt` 因前端类型缺字段而回退为按 `id` 排序，属于假排序。
- 当前可见表格列为：任务、状态、标签、AI CLI、来源；缺少可读任务编码和 AI 执行上下文。
- 当前字段不可自定义，所有用户看到同一列布局。

### 参考产品取舍
- Giraffe 可参考点：任务编码作为任务身份锚点、列表字段可支撑快速扫描、详情页结构化展示任务内容。
- Giraffe 不采用点：负责人、处理人、待我处理、@我、逾期、SLA 等团队协作字段。
- Zeus 差异化：字段围绕“我与 AI 的协作状态”设计，而不是围绕“人和组织流转”设计。

### 数据契约设计
#### SQLite 迁移
- `tasks` 表新增：
  - `task_code TEXT`：用户可读任务编码。
  - `task_sequence INTEGER`：项目内自增序号，用于生成稳定编码。
- 索引建议：
  - `idx_tasks_project_task_code(project_id, task_code)`，支持项目内唯一查询。
  - `idx_tasks_project_sequence(project_id, task_sequence)`，支持生成下一序号和回填校验。
- 唯一约束口径：项目内 `task_code` 唯一；跨项目允许相同序号但通过项目前缀或项目上下文区分。
- 回填策略：旧任务按 `created_at ASC, id ASC` 为每个项目生成序号，编码格式默认 `ZEU-000001`。若后续需要项目短前缀，可只调整渲染层，不破坏序号。
- 迁移幂等：旧库多次启动不得重复改写已有 `task_code`；只为缺失编码的任务补齐。
- 回滚：保留新增列不影响旧代码读取；如需逻辑回滚，可停止在 API 和 UI 暴露 `taskCode`，数据库列可留存。
