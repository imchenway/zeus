# TASK_20260627_001_Zeus钥匙串反复授权提示

## 结论

2026-06-27 现场截图中的弹窗不是 Zeus 自己在索取业务密码，而是 macOS Keychain 在要求授权 Zeus 读取 Electron/Chromium 的 `@zeus/desktop Safe Storage` 密钥。该密钥用于浏览器 profile 的安全存储加解密，项目显式实现的 Bot Token / API Key Keychain 服务名是 `Zeus`，与截图中的 `@zeus/desktop Safe Storage` 不是同一个来源。

当前更可能的原因有两类：

1. 用户每次点击的是「允许」而不是「始终允许」：macOS 只授权本次访问，下次启动仍会询问。
2. 当前 Zeus 打包产物是未正式签名的本地 ad-hoc 签名：`electron-builder.yml` 中 `identity: null`，当前 `dist/mac-arm64/Zeus.app` 的 `codesign` 结果为 `Signature=adhoc`、`TeamIdentifier=not set`。如果每次重新打包或二进制 hash 变化，Keychain 可能不把它视为同一个可信应用，即使曾经点过「始终允许」也可能再次提示。

## 契约变更

本次仅调查与解释，没有修改源码或运行时契约。

如后续修复，推荐契约如下：

1. 桌面 App 应拥有稳定显示名和稳定用户数据目录，避免 `@zeus/desktop` 暴露给用户。
2. 生产或长期日用构建应使用 Developer ID 签名和 notarization，不能依赖 ad-hoc 签名承担长期 Keychain 信任。
3. 安全设置页应解释：Safe Storage 是 Electron profile 密钥，Zeus 自定义 SecretStore 是服务名 `Zeus` 的独立 Keychain 项。

## 修复选项

| 选项 | 做法 | 优点 | 缺点/风险 | 回滚 |
| --- | --- | --- | --- | --- |
| A. 用户临时处理 | 在同一个未变更 app bundle 上输入 macOS 登录钥匙串密码并点「始终允许」 | 最快，无需改代码 | 重新打包或 app hash 改变后仍可能再问 | Keychain Access 中移除该 item 授权或删除 item |
| B. 清理旧 Safe Storage item | 删除 `@zeus/desktop Safe Storage` 后重启 Zeus 重新生成 | 可修复旧 ACL 异常或历史包名残留 | 会丢失 Electron profile 内依赖该密钥加密的数据；需避免误删 Zeus 自定义 `Zeus` 服务密钥 | 从备份恢复 Keychain 或重新配置相关登录/状态 |
| C. 稳定 app 名称/profile | 在 Main 进程早期锁定 app name / userData 为 `Zeus`，并设计一次性迁移 | 用户看到的 Keychain 名称更合理 | 会触发一次迁移和一次新授权；需验证旧数据迁移 | 恢复旧 userData 路径并保留旧目录 |
| D. 正式签名/公证 🌟 | 配置 Apple Developer ID 签名与 notarization，停止日用未签名/ad-hoc 包 | 从根上给 Keychain 一个稳定可信身份，也是发布正解 | 需要 Apple Developer 账号、证书、CI/本机凭据管理 | 回退到 unsigned 本地包，但提示可能恢复 |

## 实施顺序（如用户要求修复）

1. 先确认用户实际点击的是「允许」还是「始终允许」，以及是否每次启动前都重新打包。
2. 非侵入验证当前签名和 Keychain 自动锁定状态，不读取任何密钥明文。
3. 优先给用户临时规避：同一包点「始终允许」。
4. 若仍复现，提供清理旧 Safe Storage item 的明确手动步骤，强调只删 `@zeus/desktop Safe Storage`，不要删 `Zeus` 服务下的 Bot/API/database secret。
5. 若进入代码修复，先补 TDD：锁定 app name/userData 迁移、release 状态文案、打包签名检测。
6. 最终正解是签名/公证链路；这涉及外部 Apple 凭据，必须单独确认。

## 风险与回滚

- 风险：误删 Keychain 可能导致 Telegram Bot Token、外部 API Key 或数据库密码需要重新配置。规避方式：只处理 `@zeus/desktop Safe Storage`，不要处理服务名为 `Zeus` 的条目。
- 风险：改 `userData` 路径会影响 SQLite、配置、日志路径。必须设计迁移和回滚。
- 风险：正式签名/公证引入外部证书和 CI secret 管理，属于高风险发布链路变更。
- 回滚：保持当前 `identity: null` + ad-hoc 签名可恢复本地开发分发，但反复 Keychain 授权提示可能继续存在。


## 2026-06-27 实施记录：按用户决策移除 Safe Storage 钥匙串申请

### 用户决策

用户明确要求：“无论如何，移除这个申请的动作”。本轮按该决策处理，不再继续推荐“始终允许”作为主方案。

### 实现方式

| 文件 | 改动 |
| --- | --- |
| `apps/desktop/src/main/main.ts` | 在 Electron app ready 前追加 `app.commandLine.appendSwitch('use-mock-keychain')`，仅 macOS 生效，用 Chromium mock keychain 替代系统 Keychain，避免触发 `@zeus/desktop Safe Storage` 读取授权弹窗。 |
| `apps/desktop/test/main-runtime.test.ts` | 新增回归测试，锁定该开关必须出现在 `app.whenReady()` 之前，并带有明确中文注释说明代价。 |

### 影响边界

- 受影响：Electron/Chromium profile 的 Safe Storage 加解密策略；旧 profile 中依赖系统钥匙串加密的浏览器态可能不可读或重建。
- 不受影响：Zeus SQLite 项目/任务/图谱主数据。
- 不受影响：Zeus 自定义 SecretStore 服务名 `Zeus` 下的 Telegram Bot Token、外部 API Key、数据库密码；这些仍由 Zeus 安全设置显式管理。
- 不新增依赖、不改数据库、不改 API 契约、不改 renderer UI。

### 风险与回滚

- 风险：Chromium profile 内旧的 encrypted cookie/session/local storage 可能无法用 mock keychain 解密，需要重建；Zeus 当前核心数据不依赖这些浏览器态。
- 风险：这是“按用户要求禁用申请”的体验优先方案，不等价于正式签名/公证方案。
- 回滚：删除 `disableChromiumSafeStorageKeychainPrompt()` 及调用，并移除对应测试断言，重新打包即可恢复系统 Keychain Safe Storage 行为。
