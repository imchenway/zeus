# TASK_20260627_003_Zeus任务列表批量选择与列对齐

## 需求来源

用户反馈：

1. 任务列表页需要可以批量选择、批量管理。
2. 任务列表的列头和下面字段没有对齐。

本任务已进入 develop 阶段并完成源码实现；详细实现记录见文末。

## 现状确认

### 功能现状

- 入口：左侧项目 source-list 展开后进入「任务」工作区。
- 当前首层：紧凑 toolbar、任务表格、行点击打开任务详情抽屉。
- 当前批量能力：无批量选择、无批量操作栏。
- 当前字段配置：有「字段」popover，可开关列和移动列顺序。
- 当前详情管理：单个任务可在详情抽屉里运行、暂停、继续、重试、完成、取消。

当前 `TaskWorkspace.tsx` 的任务行是：

- `button.task-list-row.task-table-row role="row"`
- 行点击打开详情抽屉。
- 行内每个字段是 `span role="gridcell"`。
- 列头和行都通过同一个 `taskTableGridStyle` 写入 `gridTemplateColumns`。

这带来两个问题：

1. 批量选择需要行内复选框，但 HTML 不允许在 `button` 内嵌交互控件，继续叠加会让可访问性、点击冒泡和键盘行为变乱。
2. 列头和行虽然共享模板，但每一行都是独立 grid，且行本体是 button，旧列 class 还带有 inline-flex/grid 等覆盖，容易出现字段与列头的视觉错位。

## 根因判断

- 批量选择缺失的根因：任务列表仍以「整行按钮打开详情」为主模型，没有进入真正的「可选择数据表」模型。
- 列不齐的根因：列宽模板分散在每个独立 header/row 的 inline grid 上，缺少统一表格轨道；行按钮默认控件语义和列内旧样式进一步放大错位。
- 两者应一起修：如果只补一个批量操作栏，会和现有整行按钮冲突；如果只调 CSS，批量选择仍无法产品化。

## 需求目标

- 用户可以在任务表格中选择单项、多项、全选当前筛选结果。
- 选择后出现低噪音批量管理栏，不把功能按钮常驻堆在 toolbar。
- 批量操作必须遵守任务状态机，不允许非法状态迁移。
- 列头、选择列和行字段必须稳定对齐，字段配置后仍对齐。
- 保持任务页主路径：列表选择任务；详情抽屉查看单任务完整信息。

## 方案对比

| 方案 | 做法 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| A. 最小补丁 | 保留整行 button，在左侧伪造 checkbox 视觉 | 改动少 | HTML 交互非法，复选框与行点击冲突，后续更难维护 | 不推荐 |
| B. 前端批量编排 + 真正可选择表格 | 行从 button 改为 grid/table row，选择列独立；批量动作复用现有单任务 API，前端限并发执行 | 不改 DB/API，能一次解决批量选择和列对齐，风险可控 | 需要调整任务页 DOM 和测试 | 🌟推荐 |
| C. 新增后端 batch API | 增加 `/api/tasks/batch`，后端统一处理批量状态、归档、删除 | 后端可审计一个 batch，适合大规模任务 | 改 API 和服务端，迁移/错误码/兼容成本更高 | 后续任务量大时再做 |

## 推荐方案 B：前端批量编排 + 可选择任务表格

### Visual thesis

任务页应像 macOS 原生任务工作台：表格轨道稳定、选择模式低噪音、批量操作只在需要时出现。

### Content plan

- 首层保留：项目计数、搜索、筛选、排序、标签、新任务、任务表格；字段配置属于低频表格偏好，不能夹在筛选与“新任务”主路径之间。
- 选择后出现：已选择 N 项、全选当前结果、清除选择、批量状态修改、批量删除。
- 危险操作收纳：批量删除必须二次确认，并显示将删除的任务数量；不把删除伪装成普通按钮。
- 详情仍在抽屉：单任务详情、事件、Runtime、Git 证据不进入表格首屏。

### Interaction thesis

- 单击复选框选择，不打开详情。
- 单击行内容仍打开详情；键盘焦点在行上时 Enter 打开详情，Space 在复选框上切换选择。
- 批量操作执行时显示 `aria-live` 状态，完成后清除成功项选择，失败项保留选择并显示失败计数；删除成功项从当前列表消失。

## 业务规则

### 选择规则

- 默认不进入批量模式，表格左侧始终有低噪音选择列。
- 表头 checkbox：
  - 当前筛选可见任务全未选：未选中。
  - 当前筛选可见任务全选：选中。
  - 只选一部分：mixed。
- 「全选当前结果」只选择当前项目 + 当前筛选结果，不跨项目、不选归档列表。
- 切换项目、任务刷新或筛选变化时，自动剔除当前不可见或已不存在的 selected ids。

### 批量动作

| 动作 | 适用任务 | 后端/API | 备注 |
| --- | --- | --- | --- |
| 批量状态修改 | 按 `packages/task-core/src/index.ts` 状态机逐项判断 | `updateTaskStatus(taskId, targetStatus)`，必要时优先使用专用运行控制 API | UI 提供目标状态下拉；非法迁移跳过并提示数量。 |
| 批量删除 | 默认仅允许非 `running` / 非 `waiting_confirmation` 任务 | `deleteTask(taskId)` | 高风险动作，必须二次确认；运行中或等待确认任务默认跳过，避免误删正在执行链路。 |

批量状态修改的目标状态建议限制为状态机能解释的目标：`ready`、`running`、`paused`、`waiting_confirmation`、`completed`、`failed`、`cancelled`。实际执行时逐项按当前状态过滤：例如 `running -> completed` 可执行，`ready -> completed` 跳过。

### 并发与失败

- 批量动作按本地前端队列执行，建议并发 3，避免一次性触发大量 Runtime/API。
- 每个任务沿用现有单任务接口，保留原有任务事件和审计日志。
- 任一任务失败不阻断全部；结束后显示成功/跳过/失败数量。
- 失败任务保留选择，便于用户重试或打开详情查看。

## UI/UX 设计

### 页面唯一核心目标

管理当前项目的真实任务。

### 用户当前最需要做

从任务表格中筛选、选择一批任务，并对这批任务执行状态或归档管理。

### 首层必须展示

- toolbar：项目计数、搜索、状态、排序、标签、新任务；字段配置仅保留为“新任务”之后的低噪音图标入口，不能出现可见“字段”文字按钮。
- 表格：选择列 + 任务编码 + 意图 + 下一步 + AI 执行 + 来源 + 信号 + 更新时间。
- 选择后：批量操作栏。

### 应收纳到下一层

- 单任务详情、事件、Runtime 细节、Git diff 进入右侧抽屉。
- 批量删除进入“选择后”的批量操作栏，但不常驻在默认 toolbar。
- 字段配置继续在 popover，但入口降级为图标按钮，避免和“新任务”形成两个同级文本按钮。

### 避免项

- 不把批量按钮常驻塞满 toolbar。
- 不把低频“字段”配置做成主工具栏里的可见文字按钮。
- 不新增卡片、不新增 HERO、不恢复右侧详情停车场。
- 不让 checkbox 和整行打开详情互相抢点击区域。

## 开发设计

### 前端结构

#### `TaskWorkspace.tsx`

- 新增 props：
  - `selectedTaskIds: readonly string[]`
  - `bulkActionBusy: boolean`
  - `bulkActionStatus?: { kind: 'idle' | 'running' | 'done' | 'failed'; message?: string }`
  - `onToggleTaskSelection(taskId: string, selected: boolean)`
  - `onToggleAllVisibleTaskSelection(taskIds: string[], selected: boolean)`
  - `onClearTaskSelection()`
  - `onBulkTaskStatusChange(targetStatus: TaskStatus, taskIds: string[])`
  - `onBulkTaskDelete(taskIds: string[])`
- 行结构从 `button role="row"` 改为可选择 row：
  - `div/button` 混合方案需避免嵌套交互；推荐 `div role="row"` + 独立 checkbox + 行内容点击/键盘打开详情。
  - 表格轨道通过 CSS 变量 `--task-table-grid-template` 统一到 header 与 rows。
- 新增选择列：`minmax(32px, 32px)`。
- Header checkbox 支持 mixed 状态。
- 批量操作栏只在 `selectedTaskIds.length > 0` 出现。
- 字段配置入口排在“新任务”之后，视觉上收成 `⋯` 图标，保留 `aria-label="自定义任务字段"`、Escape 关闭和焦点回到触发器。

#### `taskWorkspaceModel.ts`

- 新增选择模型：
  - `selectedVisibleTaskIds`
  - `allVisibleSelected`
  - `someVisibleSelected`
  - `bulkStatusEligibility`，按目标状态计算可处理/跳过数量。
  - `bulkDeleteEligibility`，默认排除 running / waiting_confirmation。
- 输出 `visibleTaskIds`，供全选当前结果使用。

#### `App.tsx`

- 新增 `selectedTaskIds` 状态。
- 切换项目、任务列表刷新、筛选变化后清理无效选择。
- 新增 `runBulkTaskStatusChange(targetStatus, ids)`：
  - 按状态机过滤 eligible 任务。
  - 用现有单任务 status/runtime handlers 执行。
  - 汇总成功/跳过/失败。
  - 更新 snapshot，清理成功项选择。
- 新增 `runBulkTaskDelete(ids)`：
  - 默认跳过 running / waiting_confirmation。
  - 执行前 `window.confirm`，文案包含任务数量和不可撤销提示。
  - 调用现有 `deleteTask`。
  - 删除成功项从 selection 中移除，失败项保留。

#### `styles.css`

- 新增最终覆盖：
  - `任务表格选择列与轨道对齐最终覆盖`
  - `任务批量操作栏最终覆盖`
- Header 与 rows 使用同一个 grid token，不再各自硬编码 fallback。
- 选择列固定宽度；内容列仍支持字段配置后的动态模板。
- 行 hover/focus 保持低噪音灰底。

### 后端/API/数据库

- 首轮不新增后端 API。
- 首轮不改 SQLite 表结构。
- 复用现有接口：`updateTaskStatus`、`deleteTask`；如目标状态涉及运行控制，可继续复用现有 `runTask`、`pauseTask`、`continueTask`、`cancelTask`、`retryTask` 专用接口。
- 每个任务仍产生现有任务事件和审计记录。

## 验收标准 AC

- AC1：任务表格左侧有选择列；可单选、多选、取消选择。
- AC2：表头 checkbox 可全选/取消当前筛选结果，并支持 mixed 视觉/ARIA 状态。
- AC3：选择 1 个以上任务后出现批量操作栏；未选择时不占首屏空间。
- AC4：批量状态修改提供目标状态选择；每个任务必须按状态机校验，非法迁移跳过并提示数量。
- AC5：批量删除必须二次确认；确认文案包含将删除的任务数量和不可撤销提示。
- AC6：批量删除默认跳过 `running` / `waiting_confirmation`，避免误删正在执行或等待确认任务。
- AC7：批量执行成功后刷新真实 snapshot；成功项清出选择，失败项保留选择。
- AC8：删除成功后任务从当前列表消失；失败删除项仍保留选择。
- AC9：列头、选择列和行字段在默认列、隐藏列、移动列后仍对齐。
- AC10：行内容点击/Enter 仍打开详情；点击 checkbox 不打开详情。
- AC11：键盘可访问：checkbox 可 Tab 到达，行可上下/Home/End 导航，focus 样式清晰。
- AC12：空列表和筛选无结果不显示批量操作栏，不出现空白停车场。
- AC13：不新增依赖、不改构建、不改 DB。

## 风险与回滚

| 风险 | 缓解 |
| --- | --- |
| 行不再是 button，可能影响整行打开详情 | 用 row click + Enter 键测试锁定；checkbox stop propagation。 |
| 批量状态修改部分失败造成用户困惑 | 执行结果显示成功/跳过/失败数量，失败项保留选择。 |

### 返修记录（2026-06-27 10:39 CST）：收纳“字段”文字按钮

#### 现场现象

用户截图指出任务页筛选栏中“字段”按钮夹在“标签”和“新任务”之间，视觉上像一个常用主操作，挤占主路径。

#### 影响

- “新任务”应是默认工具栏唯一高权重动作；“字段”是低频列偏好，不应以文本按钮占据同级位置。
- 继续保留该文字按钮会让任务页看起来像后台控件堆叠，也和本任务“批量动作只在选择后出现”的低噪音原则冲突。

#### 根因

上一轮为了保留列设置能力，把字段配置 popover 直接放在 compact toolbar 中，并使用可见文案“字段”。这虽然功能可用，但没有把“列偏好”降级到低频入口。

#### 修法

| 文件 | 变更 |
| --- | --- |

### 契约变更

- 前端组件契约：`TaskWorkspace` 增加可选批量选择与批量动作 props。

### 状态修改规则

- 前端先使用与 `task-core` 一致的状态迁移表过滤非法迁移。
- 对 Runtime 相关目标优先复用专用控制接口：
  - `cancelled` → `onCancelTask`
  - `paused` → `onPauseTask`
  - `running` 从 `ready` → `onRunTask`，从 `paused` / `waiting_confirmation` → `onContinueTask`
  - `ready` 从 `failed` / `cancelled` → `onRetryTask`
- 其他合法状态使用 `onUpdateTaskStatus`。
