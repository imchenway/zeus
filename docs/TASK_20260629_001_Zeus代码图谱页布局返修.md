# TASK_20260629_001 Zeus 代码图谱页布局返修

## 1. 任务来源与判断

用户基于真实截图反馈当前 Zeus 代码图谱页“简直是一堆垃圾”。截图展示的是 `/Users/david/hypha/zeus` 项目的代码页，当前页面把顶部对象栏、图谱状态、视图切换、主图谱、右侧检查器、Sigma/WebGL 预览、React Flow 预览、搜索/问答/Mermaid/节点清单工具和底部项目动作全部同时摊开，导致主路径被淹没。

本轮先做只读排查与方案确认，不修改源码。

## 2. 唯一核心目标

进入代码页后，用户应在 2 秒内看懂当前项目的真实代码图谱，并能从图谱节点继续打开源码或创建任务。

非目标：

- 不把代码页改成营销页或后台大屏。
- 不增加假节点、假任务、假图谱或假运行时数据。
- 不引入新依赖、不改构建、不改后端接口和数据库。

## 3. 截图暴露的问题

| 编号 | 现象 | 影响 | 初步证据 |
|---|---|---|---|
| P0-1 | 主画布巨大空白，真实 8 个节点缩成中间一小团，节点文字重叠 | 用户看不到图谱结构，代码页主角失效 | 用户截图；`GraphCanvas` 使用固定 SVG 视窗与固定节点尺寸 |
| P0-2 | Sigma/WebGL 与 React Flow 预览常驻在主画布下方，被压成细条 | 辅助运行时抢首屏，又不可读 | `CodeMapView` 在 `GraphCanvas` 后直接渲染 `GraphRuntimeCanvas` |
| P1-1 | 右侧检查器大量空白和重复卡片感 | 当前对象信息不聚焦，像后台属性面板 | `code-map-inspector-pane` 常驻右栏，默认同时渲染节点和边详情 |

## 4. 根因假设与证据链

### R1：信息架构退化成“能力全摊开”

- 代码：`apps/desktop/src/renderer/App.tsx` 中 `CodeMapView` 同时渲染主图谱、右侧检查器、运行时预览和二级工具区。
- 结果：页面没有单一工作台主路径，用户第一眼看到的是多个区域的堆叠，而不是图谱。
- 置信度：高。

### R2：图谱画布尺寸与可见节点规模脱钩

- 代码：`GraphCanvas` 根据服务端 `layout.width/height` 或 fallback 得到 SVG viewBox；CSS 又给 `.graph-canvas-svg` 固定 `inline-size: max(1180px, 100%)` 与 `block-size: clamp(360px, 58vh, 640px)`。
- 结果：小图谱节点按固定 24px 半径绘制在大画布里，视觉比例过小，标签容易互相压住。
- 置信度：高。

### R3：测试把旧坏布局固化成“通过”

- 已运行聚焦 baseline：
  `pnpm vitest run apps/desktop/test/app-code-map-rendering.test.tsx --testNamePattern "dominant code-page stage|runtime preview|project code graph stage|secondary inspector" --reporter=verbose`
- 结果：5 passed / 95 skipped。
- 说明：当前测试证明 DOM 顺序和 CSS 字面量存在，但没有验收“首屏是否像真实产品、图谱是否可读、低频工具是否收纳”。
- 置信度：高。

## 5. 推荐修复方案

### 方案 A：一屏图谱工作台重构（推荐）

把代码页改成“图谱画布 + 可收纳检查器 + 单行对象工具栏”的工作台：

1. 顶部只保留一个紧凑对象栏：项目名、路径、当前视图、节点/边计数、扫描/打开图谱/查看变更/更多。
2. 视图切换并入对象栏或图谱顶部细分段控件，不再占独立大行。
3. 主画布使用内容感知 viewBox：小图谱按节点边界加 padding 放大到可读尺寸；大图谱继续保留滚动/压缩保护。
4. 默认不渲染 Sigma/WebGL 与 React Flow 预览；收进“运行时预览”二级工具或更多菜单。
5. 右侧检查器只显示当前选中对象；边详情和节点详情不同时抢焦点，窄屏默认收成抽屉。
6. 搜索/问答/Mermaid/节点清单收进右侧工具 tabs 或抽屉，默认不占底部首屏。
7. 底部项目上下文 rail 在有图谱时压成一行状态，或移入顶部更多菜单，不再形成第四层区域。

优点：直接解决截图里的主路径混乱和空白问题；符合 Zeus Design Contract v2；后续可测试锁定。
缺点：改动涉及 `App.tsx` 与 `styles.css` 的结构、状态和测试，工作量比纯 CSS 调参大。

### 方案 B：小修 CSS 与画布缩放

只调整图谱缩放、隐藏运行时预览高度、压缩底部工具区。

优点：改动小。
缺点：仍保留“功能全摊开”的根因，很容易修成另一版拥挤页面。

### 方案 C：只放大节点与字体

提高节点半径、文字字号和 canvas zoom。

优点：最快。
缺点：只治小图谱不可读，不解决信息架构垃圾堆叠，不推荐。
