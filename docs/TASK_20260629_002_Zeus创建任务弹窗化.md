# TASK_20260629_002 Zeus 创建任务弹窗化

## 结论

用户反馈成立：任务页「新任务」现在是一次点击即创建默认任务，然后直接打开详情抽屉；这会让用户在确认标题、描述和标签前就产生真实任务记录，也不符合用户明确提出的「创建任务应该是弹窗，参考 giraffe」。本任务建议把任务页的新建入口改为 **Zeus 原生居中弹窗**：先收集最少必要信息，提交后再创建真实任务，成功后关闭弹窗并打开新任务详情抽屉。

## 当前现状

### 功能现状

- 入口：任务页顶部工具栏「新任务」按钮，以及空任务态里的「新任务」按钮。
- 当前行为：点击后立即调用 `createProjectDefaultTask({ openCreatedTaskDrawer: true })`。
- 用户感知：没有创建前确认，也没有填写标题/描述/标签的步骤；真实任务先落库，之后才在抽屉里编辑。
- 已有保障：创建后会清空可能遮挡新任务的本地筛选，并打开任务详情抽屉，避免“点击无反馈”。

## 视觉与交互 thesis

- visual thesis：创建任务是任务工作台的“承诺动作”，需要一个克制的 macOS 居中 sheet，而不是直接落库或右侧详情抽屉。
- content plan：首层只展示任务标题、任务说明、标签；项目路径和来源作为只读上下文；模板/模式这类低频选项默认不在首屏铺开。
- interaction thesis：点击新任务先打开弹窗；焦点进入标题；提交中禁用重复提交；成功后关闭弹窗并打开新任务详情抽屉；Esc / 取消关闭并恢复焦点。

## 推荐方案

### 方案 A 🌟：Zeus 原生创建弹窗（推荐）

点击任务页所有「新任务」入口后，打开居中弹窗。弹窗收集：

- 标题：必填，默认可填入本地化默认标题，但应允许用户直接改。
- 说明：可选，默认使用当前语言下的任务说明草稿。
- 标签：可选，逗号分隔并去重。
- 来源上下文：只读显示当前项目路径，不要求用户重复输入。

提交后通过现有 `client.createTask(...)` 能力创建真实任务，再刷新 dashboard，清理任务页筛选，选中新任务并打开任务详情抽屉。

优点：

- 符合用户明确要求。
- 不再产生“空白默认任务”后补编辑的反直觉链路。
- 不新增依赖，不改数据库。
- 保留现有创建成功后打开详情抽屉的反馈闭环。

缺点：

- 需要调整前端 App props 契约：从仅 `onCreateDefaultTask(projectId)` 扩展为可传入 draft 的创建入口。
- 需要新增弹窗焦点管理和表单验证测试。

### 方案 B：确认弹窗 + 仍创建默认任务

点击「新任务」后只弹出确认框，确认后仍创建默认任务。

优点：改动最小。
缺点：没有真正解决“创建前填写任务内容”的问题；用户体验仍像防误触确认，不像任务创建。

### 方案 C：照搬 giraffe 完整任务表单

引入大表单：负责人、协作人、排期、迭代、附件、富文本等。

优点：功能完整。
缺点：明显过重，与 Zeus 个人本地 AI 工作流不匹配；会把任务页重新带回后台表单堆叠。

## 推荐后的契约变更

### 前端契约

- 任务页「新任务」按钮不再直接调用创建 API，只打开 `task-create-modal`。
- 空任务态「新任务」按钮复用同一个弹窗入口。
- 弹窗具备：`role="dialog"`、`aria-modal="true"`、标题关联、错误提示关联、Escape 关闭、焦点恢复。
- 提交中按钮显示 loading/disabled，不允许重复提交。
- 关闭弹窗不创建任务，不污染真实数据。
- 创建成功后延续现有行为：清空本地筛选、打开新任务详情抽屉。

### API / 数据契约

- 不改 SQLite 表结构。
- 不新增后端接口。
- 前端 props 建议新增或替换为 `onCreateTask(projectId, draft)`，内部仍调用现有 `client.createTask(CreateTaskRequest)`。
- `sourceContext.path` 仍来自当前项目路径，不由用户输入。

### 多端边界

- 本任务仅影响 Zeus macOS 桌面渲染端。
- 不影响 Telegram、Runtime、Code Map、Git Diff、任务模板创建、图谱节点创建任务等入口；这些入口已有明确来源上下文，不强行改为弹窗。

## 风险与回滚

- 风险：增加弹窗后，想快速创建默认任务的用户多一步输入。缓解：标题默认预填，并让 Enter/Cmd+Enter 快速提交。
- 风险：App 当前创建任务入口只传 projectId，需要轻微调整 props 契约。缓解：仍走现有 `client.createTask`，不改服务端和存储。
- 风险：弹窗若做太大，会回到 giraffe 团队项目表单。缓解：只保留标题/说明/标签/只读项目来源。
- 回滚：回退新增弹窗组件/样式/测试，将 TaskWorkspace 的 `onCreateTask` 重新接回 `createTaskFromTaskToolbar`；数据库和接口无迁移成本。

- 前端 draft：
  - `title: string`，必填，不提供默认值。

## 2026-06-29 PLAN：粘贴后附件预览方案

### 需求理解

用户已确认图片/本地图片文件可以正常粘贴进创建任务弹窗。下一步不是继续修“能不能粘贴”，而是解决“粘贴后我能不能确认粘贴的是哪张图、能不能删除、创建后还能不能看见”的预览体验。

根本目标：任务说明仍保持纯文本输入，图片/文件作为本机附件证据进入任务上下文；用户粘贴后立即获得可视确认，不需要根据长路径猜测是否粘对。

### 当前现状

- 创建任务弹窗已有 `任务说明` textarea、`图片与附件`区域、`添加图片或附件`按钮。
- 粘贴或选择附件后，`TaskCreateFormState.attachments` 只保存 `{ path, name, kind }`。
- 当前 UI 只显示附件名、类型和完整路径，不显示图片缩略图，也没有选中预览、加载失败态或创建后详情预览。
- Renderer CSP 已允许 `img-src ... file:`，因此本地图片路径可作为预览来源，但需要做 URL 编码和加载失败降级。
- Task Detail 抽屉也只显示附件文本行，创建后无法视觉确认截图内容。

### UI/UX 设计判断

- visual thesis：克制的 macOS 本地附件托盘，像 Finder Quick Look 的轻量确认，而不是富文本编辑器。
- content plan：说明输入框保持主路径；其下方出现“附件预览托盘”；图片显示缩略图和选中预览，文件显示文件行；项目来源继续在底部作为事实信息。
- interaction thesis：粘贴后附件托盘轻量出现；点击缩略图切换主预览；删除按钮只删除当前附件；图片加载失败时降级为文件证据行。

### 方案对比

| 方案 | 描述 | 优点 | 缺点 | 判断 |
| --- | --- | --- | --- | --- |
| A. 内联附件预览托盘 | 在任务说明下方显示选中图片预览 + 附件队列，创建后详情抽屉复用缩略图预览 | 不破坏纯文本输入，不嵌套弹窗，粘贴后立即可确认，符合 macOS 产品工具 | 弹窗会稍微变高，需要控制最大高度 | 推荐 |
| B. 仅缩略图队列 | 图片只显示 56px 缩略图，不做大预览 | 改动最小，布局稳定 | 截图内容看不清，用户仍可能点错 | 适合后续压缩模式，不作为主方案 |
| C. 点击后 Quick Look 大预览 | 附件行点击打开大图预览层 | 图片看得最清楚 | 创建任务弹窗里再开预览层，焦点和关闭逻辑复杂 | 后续增强，不做 V1 |
| D. 富文本说明内嵌图片 | 图片直接出现在任务说明输入框里 | 看起来像聊天输入框 | 需要富文本编辑器，改变任务 description 数据结构，风险高 | 不推荐 |

### 推荐方案 A：内联附件预览托盘

#### 主路径

1. 用户在任务说明中粘贴图片或选择本机图片。
2. 系统保存或引用本机路径，合并到 `attachments`。
3. 如果存在图片附件，显示一块选中预览区：图片等比缩放，最大高度约 160px，文件名和“本机附件”元信息在旁边或下方。
4. 附件队列展示所有附件：图片用缩略图，非图片用文件图标/扩展名块。
5. 用户可点击缩略图切换选中图片，可点删除移除附件。
6. 提交后，任务详情抽屉在“图片与附件”区展示同样的缩略图证据，避免创建后只剩路径。

#### 组件与数据设计

- 新增前端组件建议：`TaskAttachmentPreviewList`。
- 输入：`attachments: TaskCreateAttachment[]`、`mode: 'editable' | 'readonly'`、`onRemove?: (path) => void`。
- 不新增数据库字段，不改 API。继续使用 `{ path, name, kind }`。
- 新增 helper：`toTaskAttachmentPreviewSrc(path)`，把绝对路径编码成 `file://` URL；仅 `kind === 'image'` 时渲染 `<img>`。
- 维护本地 UI 状态：`selectedAttachmentPath`、`failedPreviewPaths`。
- 对外边界：图片加载失败不阻断创建任务，降级显示“无法预览，本机路径已保存”。

#### 布局规则

- 创建任务弹窗：任务说明下方直接接附件预览托盘，不把所有图片塞进说明框里。
- 图片预览：单张主图最大高度 160px，`object-fit: contain`，背景为冷中性弱底。
- 多附件：最多显示 4 到 6 行自然高度，超过后附件队列内部滚动，不让弹窗无限长。
- 390px 窄屏：主预览在上，附件队列在下，删除按钮保持 28px 以上点击面积。

#### 可访问性与状态

- 附件区使用 `<section aria-label="图片与附件">`，列表使用 `<ul>`。

## 2026-06-30 DEVELOP：Finder 复制本地图片时保存成文件图标的二次修复

### 现象

- 用户在任务附件放大预览中看到的是 1024×1024 的 `JPG` 文件图标，而不是原始图片内容。
- 该附件已经被保存到 Zeus 的 `userData/task-attachments` 目录，但保存进去的二进制本身就是 Finder 图标 PNG，因此旧附件无法从保存文件中反推出原图。

### 根因

- macOS Finder 复制本地图片文件时，Electron 可能只暴露 `text/plain` / `text/uri-list` 两个格式；其中 `readText()` 只有文件名，`readBuffer('text/uri-list')` 为空。
- 同时 `clipboard.readImage()` 可能返回 Finder 文件图标位图，而不是文件内容。
- 旧修复只覆盖了 Electron 能读出 file URL 的情况；没有覆盖 Finder 只把真实路径放在 macOS `furl` pasteboard class 里的情况。

### 修法

- Main 进程在读取剪贴板附件时，额外通过 `/usr/bin/osascript` 读取 macOS pasteboard 的 `«class furl»`，拿到真实 POSIX 文件路径。
- `readTaskClipboardAttachmentsFromClipboard(...)` 新增系统文件引用入口，并继续保持优先级：真实本地文件路径 > inline data image > readImage bitmap > 原生图片 buffer。

## 2026-06-30 返修记录：新建对话输入空值与粘贴兜底

### 现象

新建对话空态输入框在打开后可能带入上一条任务说明；用户删除这段预设值后再粘贴附件，界面进入安全暂停错误页。另一个表现是从 Finder / Paste.app 复制本机图片或文件后，在该输入框内粘贴没有明显反应。

### 根因

- Electron hydration 同步真实 snapshot 时，会把当前任务的 `description` 再写回 `taskEditForm`；新建对话空态也复用了这份表单状态，因此草稿被旧任务说明污染。
- 会话粘贴处理在 `await` 之后才读取 `event.currentTarget`，React 事件对象跨异步边界后可能失效，导致清空输入后粘贴触发界面错误。
- 原生剪贴板读取失败时，渲染端旧逻辑直接返回“已处理”，阻断了 DOM paste 事件中的文件回退链路；同时会话输入框缺少和任务创建弹窗一致的 `Cmd/Ctrl+V` 原生剪贴板兜底。

### 修复

- snapshot 同步时，若当前处于新建对话草稿态，不再用任务详情覆盖输入框内容；新建对话输入保持空值，只显示 placeholder。
- 会话粘贴处理在异步调用前捕获 textarea 控件，文字粘贴恢复光标不再依赖失效事件对象。
- 原生剪贴板保存失败时返回未处理，让 DOM 文件列表继续兜底；文件读取异常只展示附件粘贴错误，不再让错误边界接管整页。
- 新建对话与已有会话 composer 都补齐 `Cmd/Ctrl+V` 延迟兜底：如果 Electron 没有派发 DOM paste 事件，就让 Main 进程读取 macOS 原生剪贴板附件。

### 回归用例

- `keeps the zero-state new chat input empty across snapshot hydration and async paste`：锁定新建对话输入框不被 snapshot 任务说明污染、粘贴控件必须在异步前捕获、并且有原生剪贴板兜底。
- 既有 `zero-state new chat attachments`、`conversation`、`attachments`、`clipboard` 相关测试继续覆盖附件轨、托管预览、聊天记录展示、任务创建与任务详情入口。
