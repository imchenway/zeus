## 结论

我会把 Zeus 的设计从「功能模块集合」继续收敛成「项目作战室」：围绕真实仓库扫描后的代码图谱建立 Aha 主轴，再把任务、Runtime、Git Diff、Telegram 和设置都变成图谱后的下一步，而不是彼此并列抢首屏。

## 推荐方案

### 🌟 方案 A：项目作战室主线，推荐

- 首屏主路径：选择真实仓库 → 扫描 → 代码图谱 → 从节点打开源码/创建任务 → Runtime 执行 → Git Diff 审查。
- 左侧只承担项目和当前项目内入口，右侧工作区一次只服务一个核心目标。
- 代码图谱页继续做最强主舞台，低频工具收进 inspector / popover / drawer。
- 任务、会话、设置都用同一套 source-list、object-toolbar、composer、decision-rail、graph-canvas 语义 token。
- 适合当前产品定位，风险可控，可用现有测试契约逐页收口。

### 方案 B：只做视觉抛光

- 优化颜色、边框、圆角、间距和控件态。
- 优点：快。
- 缺点：不解决「用户为什么从图谱继续行动」这个核心问题，容易变成好看的后台页面。

### 方案 C：新手向欢迎/向导优先

- 做强首次启动、配置、扫描向导。
- 优点：降低第一次使用门槛。
- 缺点：如果图谱和任务主路径还不够清楚，向导结束后仍会迷路。

## 优化路线

1. **先固化 Aha 首屏**：Dashboard / Projects 不追求 KPI 卡片，直接引导「选择真实仓库、扫描、看图谱」。
2. **代码图谱继续做主角**：图谱画布占首屏，inspector 只解释当前对象，工具按需展开。
3. **任务页变成执行队列**：任务列表默认表格化，批量、列偏好和详情抽屉只在需要时出现。
4. **会话页变成项目上下文输入面**：composer 保持开放输入，但始终展示当前项目、Runtime、附件和下一步。
5. **设置页改成配置健康检查**：按「必须配置 / 可选增强 / 危险确认 / 发布状态」重新组织，不只是 tab 表单。
6. **建立设计门禁**：每页用截图和 DOM 测试证明唯一核心目标、真实状态、键盘路径、窄屏可读、无旧样式回流。

## 不建议做的方向

- 不建议改成营销式 hero、渐变大屏或指标卡片首页。
- 不建议继续堆同款卡片、重复标题和后台表单面板。
- 不建议为了显得热闹而加入假项目、假任务、假图谱或假 AI 回复。
- 不建议只调色或只换圆角，因为 Zeus 当前价值主要来自「真实本地证据链」而不是装饰。
