需求决策：Zeus 最新页面重构建议，保留克制骨架，重建“当前对象 → 下一步行动”

我的判断不是推翻当前 UI。最新截图说明 Zeus 已经有一套接近 macOS source-list 的基础语言、真实空态和代码图谱核心；下一步应该做信息架构重构，把任务、代码、会话、设置从四个页面孤岛收敛为同一套项目工作台语法。

总体健康度：可用，但主路径还没聚焦

当前版本最大问题不是视觉脏，而是“页面都在，但每页都像单独工具”。用户需要一眼知道当前项目、当前对象、状态和下一步。

优先级问题与修法

我会按“先结构、再页面、最后微交互”的顺序处理。这样风险最低，也最符合当前 Zeus 已有的设计契约。

P0统一 object-toolbar 与 context-inspector

所有项目内页面顶部都应该稳定展示项目、当前对象、状态、主操作。右侧详情统一成 inspector，而不是每页单独设计一套详情面板。

  • 代码页：当前节点、边、来源、创建任务。
  • 任务页：当前任务、状态机、AI CLI/Runtime/Git 证据。
  • 会话页：当前会话、任务绑定、环境状态、后续输入。

P1把 Code Map 作为项目作战室的主舞台

代码图谱已经是 Zeus 的 Aha Moment。它不应只是“代码页里的一个视图”，而应成为项目理解、创建任务和审查变更的入口。

  • 放大图谱节点与关系，减少底部来源 chips 的视觉重量。
  • 视图切换收进 mode-rail，默认突出当前图类型和真实来源。
  • 选中节点后提供“打开源码 / 创建任务 / 问图谱”。

P1任务和会话合并成一条执行线索

现在任务详情、会话消息、环境信息、CLI 推送分散在不同区域。建议把它们统一成“时间线 + 命令 composer + 环境 inspector”。

  • 会话不是独立聊天，而是任务执行的沟通层。
  • 任务详情默认右侧 inspector，不默认全屏抽屉。
  • 底部 composer 绑定当前任务和当前环境。

P2设置页改成状态驱动的配置工作台

设置页不需要变花，但要显示外部等待项和安全状态。把 AI CLI、Telegram、Git 确认、发布更新等从“标签页”改为带状态的配置分组。

  • 每组显示：状态、影响范围、主要动作、最后验证时间。
  • 保存按钮改为 sticky footer 或组内明确 CTA。
  • 未配置项不伪造成功，只显示下一步。

推荐重构后的三个页面 Shell

这三个 Shell 是设计和前端结构都能复用的骨架。它们不会改变 Zeus 的本地优先原则，也不需要引入新视觉品牌。

1. Project Workspace Shell

  • 用于 Code Map、Sessions、Runtime、Git Diff。
  • 中心是 stage，右侧 inspector，顶部 object-toolbar。
  • 让“从图谱到任务”成为跨页动作。

2. Work Queue Shell

  • 用于 Tasks 与任务详情。
  • 表格只在有足够任务时占主导；少量任务时转为列表 + inspector。
  • 风险动作放在 decision-rail，保持确认与审计。

3. Configuration Shell

  • 用于 Settings、项目设置、安全配置。
  • 每个配置组先显示状态，再显示输入项。
  • AI CLI / Telegram / Git / 发布等待项成为一屏可扫的状态清单。

可访问性与状态验收

截图只能发现风险，不能证明完整合规。若后续开发，应把下面几项作为每个切片的验收门禁。

交互与辅助技术

  • 所有 icon-only button 必须有稳定可读名称。
  • source-list、tabs、table、drawer、modal、composer 都要有可预测 Tab 顺序。
  • drawer/modal 打开后聚焦进入，关闭后回到触发器；Esc 可关闭。
  • 状态不只靠颜色表达，必须有文本标签。

状态与真实数据

  • loading 使用局部骨架，不遮住已有真实数据。
  • empty 说明这里会出现什么、为什么重要、下一步怎么做。
  • error 说明影响范围和恢复动作，不暴露敏感堆栈。
  • external wait 明确 AI CLI、Telegram、发布凭据等等待项，不伪造成功。
