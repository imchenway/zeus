# TASK_20260702_002 Zeus 任务表格与新会话快捷键修复

## 原始问题

用户在任务页截图中指出三项问题：

1. `更新时间` 列异常：同一个单元格同时显示格式化时间和原始 ISO 时间，导致列内容异常、行高被撑高。
2. `Cmd+N` 会新开窗口：不符合期望，应改为开启新会话。
3. 任务表格行高太高，列需要可以自定义。

随后用户连续补充现场截图，指出以下二次 / 三次回归：

- 默认列仍出现横向滚动；`上下文来源` 把项目路径显示在表格行内继续撑高行。
- 自定义列弹层过高，宽度控件换行，并且被任务表格首行 / 横向滚动条所在容器裁切。
- 横向滚动条不应贴在首行下方。
- 2026-07-02 05:49 现场截图继续反证：上一版把滚动条挪到底部时强制制造了大块空白滚动视口，首屏表格反而变成“单行 + 半屏空白 + 底部滚动条”。
- 2026-07-02 后续现场继续反证：再上一版为了避免滚动条，把默认表格右侧列视觉裁掉，看起来像直接不展示剩余列；这不是列未渲染，而是右侧列不可达。

本轮遵循用户边界：除按用户授权退出 Zeus 做打包门禁外，不做 GUI 点击、截图或人工打开复核；修复基于代码、自动化测试、构建和打包证据。

## 根因判断

- 时间列异常的根因是模型层把原始 ISO 时间当作 secondary 文本继续输出到表格单元格。
- `Cmd+N` 异常的根因是原生菜单仍保留窗口级 `newWindow` 行为，没有路由到 Renderer 的新会话动作。
- 行高和列自定义问题来自三处叠加：默认列轨道过宽、来源路径进入默认行、字段弹层既高又把列宽控件排到第二行。
- 弹层裁切问题来自父级 / 列表滚动容器的 `overflow` 边界。
- 05:49 空白回归的根因是错误地用 `block-size: min(72vh, 760px)` 和列表 `block-size: 100%` 去“安置滚动条”，这会在单行任务时人为制造巨大空白。
- 最新列不可见回归的根因是把非空任务列表恢复为 `overflow: visible` 后，没有保留横向可达兜底；默认 7 列其实仍在 DOM / table 结构里，但右侧 `证据`、`更新时间` 会被工作区视口裁掉，视觉上等价于隐藏剩余列。

## 设计口径

- 时间列只显示一行可读时间，原始 ISO 值不再作为任务表格单元格副文本展示。
- `Cmd+N` 从窗口级动作改为会话级动作：Main 进程恢复主窗口后通过 preload 通知 Renderer 打开新会话草稿。
- 列自定义从“只可显隐 / 排序”扩展为“显隐 / 排序 / 窄中宽”，入口显示为“自定义列”。
- 行密度回到 macOS 列表密度，但保持 44px 命中区。
- 默认列轨道继续收敛，不用 `920px` 硬最小宽度；上下文路径保留为搜索来源，不进入默认表格行。
- 自定义列弹层保持短视口高度，字段名、上 / 下移按钮和窄 / 中 / 宽控件同一行。
- 自定义列弹层必须能越过任务表格列表边界显示；父级不裁切弹层。
- 默认 7 列必须完整渲染，且右侧 `证据`、`更新时间` 不能因为“避免滚动条”而不可达。
- 非空任务列表保持内容高度：`block-size: auto`，禁止恢复 `block-size: min(72vh, 760px)` 或列表 `block-size: 100%` 这类撑高滚动区方案。
- 若默认宽度不足，允许横向滚动；滚动条只跟随表格内容，不把列表撑成 72vh 空白。
- 宽列 / 扩展字段仍保留 `task-list-horizontal-scroll` 标记和横向滚动行为。
