页面设计稿：Zeus 任务页工作台优化草图

这张图只画方向，不改代码。事实来源是当前 Zeus 任务页截图：左侧项目 source-list 已稳定，右侧任务页现在主要问题是表格过窄、右侧列依赖横向滚动、筛选栏与表格关系弱、大片空白让首屏显得不完整。

推荐先走“紧凑表格优先”：它最贴近现有信息架构，改动成本低，能直接解决你截图里的视觉问题。另两个方向用于对照：一个更强调工具栏分组，一个引入轻量详情轨道。
维度
紧凑表格优先 推荐
分组工具条优先
轻量详情轨道
核心想法
把任务表格作为主角，压缩工具栏与列宽，让默认 7 列尽量首屏可读。
把筛选、排序、标签、列设置合成一条对象工具栏，减少顶部控件散落感。
保留表格，但右侧提供选中任务的轻量上下文，减少每列承载压力。
优点
最少改动；最符合当前“纯表格工作区”；能立刻减少空白和滚动焦虑。
顶部秩序更强；后续新增批量操作或视图切换更自然。
任务详情更近；长字段可移出表格，表格可更紧凑。
风险
极窄窗口仍需要横向滚动兜底。
如果控件过多，工具栏会变成第二个拥挤区域。
违反“任务页右侧单工作区”的既有边界风险，需要谨慎。
调整成本
低：主要是 CSS grid、工具栏密度、列宽策略和测试。
中：需要整理 toolbar DOM 与响应式。
中高：需要定义选中态、详情轨道、键盘焦点和收纳规则。

响应式状态板

桌面、平板、窄屏，以及页面状态约束

桌面：完整 7 列优先可读

平板：保留关键列，次要列可滚动

窄屏：卡片化行，不依赖横向拖动理解

默认

真实任务直接进入表格扫描。

加载

局部骨架，不盖住侧栏。

空

没有任务，主操作是新任务。

错误

说明影响范围与恢复动作，不暴露堆栈或密钥。

主路径页面流

从看任务到调整列，不再让滚动条成为主角

进入任务页

左侧 source-list 保持现状；右侧首屏直接看到项目名、筛选和任务表格。

扫描任务

表格默认 7 列按重要性压缩；行高 44px；状态和更新时间可直接识别。

筛选 / 排序

搜索、状态、排序留在顶部；标签和低频设置可收纳，避免工具栏散乱。

调整字段

自定义列弹层继续不被裁切；宽列和扩展字段可触发横向滚动。

打开详情

深层事件、AI 推送、Git diff 进入任务详情，不挤占默认表格首屏。
