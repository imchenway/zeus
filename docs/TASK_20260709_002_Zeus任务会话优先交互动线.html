核心结论：Zeus 的实现入口是本机 app-server；本次已让任务运行先由 app-server 创建持久化 conversation 并写入首条 task_prompt，再把 Runtime/CLI 作为会话下的执行引擎挂载；前端文案和禁用态也改为“创建 app-server 会话”，不会再因 AI CLI 未配置或 packaged App 的 app.asar 根目录误判而阻断会话创建；成功创建后会直接切到对应 Sessions 会话，事件时间按本机本地时间展示到秒；已有会话输入框输入/删除不再触发错误边界，发送后续消息会通过 app-server 写入 conversation message 并转发到绑定 Runtime；如果旧 Runtime 已退出、丢失或当前 app-server 进程找不到，会自动续接新的 Runtime 并回填同一个 conversation；Runtime stdout/stderr 也会镜像为 conversation message，因此用户能在会话页看到真实模型输出。

交互修复口径

  • 已实现：每次任务运行先通过 app-server 创建持久化 conversation。
  • 已实现：任务详情主按钮不再因为 AI CLI 未配置而灰掉；Finder/Dock 启动时也会补入常见本机 CLI 路径。
  • 已实现：packaged App 的 projectRoot=app.asar 不再阻止注册项目 localPath 作为 Runtime 工作目录。
  • 首条消息来源建议标记为 task_prompt，metadata 记录 taskId、projectId、adapter、prompt 生成版本。
  • Runtime 成功后把 runtimeSession.id 回填 conversation；失败则保留 conversation 并写失败消息。
  • 已有 conversation 的后续消息如果遇到旧 Runtime 不可写，会保留用户消息、续接新 Runtime、回填新的 sessionId，不再把 AI Runtime session not found 作为终点。
  • Runtime stdout/stderr 会镜像到绑定任务 conversation：stdout 是 assistant/runtime_stdout，stderr 是 system/runtime_stderr，metadata 记录 runtimeLogId 便于去重和追溯。
  • 会话页应作为用户主画布，不再让任务页承担长对话状态。
