Codex 如何续接已经停止运行的旧会话

Codex 不为每个会话长期保留一个模型进程。稳定身份是 threadId，持久事实是磁盘 rollout；旧会话再次发送消息时，app-server 先恢复或复用 thread，再在同一 thread 上创建新的 turn。

同 thread 连续多轮已实测 跨 app-server 进程 resume 已实测 没有 threadId/rollout 就不能真实 resume

停止后再次发送消息的真实顺序

1 · 用户打开旧任务客户端从持久化关系取回 provider threadId。
→
2 · 判断 thread 状态已加载且 idle：直接复用；未加载或 app-server 已重启：调用 thread/resume。
→
3 · 从 rollout 恢复app-server 按 threadId 读取 ~/.codex/sessions/.../rollout-...jsonl，恢复 turns、上下文与配置。
→
4 · 新建本轮客户端发送 turn/start：threadId 不变，产生新的 turnId 与 clientUserMessageId。
→
5 · 流式执行并回 idle接收 item/delta/completed；turn/completed 后 thread 回 idle，之后仍可继续下一轮。

两种“停止”的差异

Turn 已结束，但 app-server 仍运行

  1. thread 只是 idle，没有会话专属进程需要重启。
  2. 客户端可直接在同一 threadId 上调用 turn/start。
  3. 每条新消息获得新的 turnId。

Codex/App-server 整体已退出或 thread 未加载

  1. 新 app-server 进程启动并完成 initialize。
  2. 优先用 threadId 调用 thread/resume；普通客户端不应依赖不稳定的 rollout path。
  3. resume 返回同一 thread 及已恢复 turns，再调用 turn/start。

什么可以续接，什么不能

历史类型持久事实再次发送真实性边界
已完成过真实 turn 的 native threadthreadId + rollout JSONLthread/resume → turn/start真实续接，同一 threadId
同进程内 idle threadapp-server 内存状态 + rollout直接 turn/start真实续接，同一 threadId
只调用过空 thread/start可能尚无物化 rolloutresume 可能返回 no rollout found不能宣称可恢复
Zeus legacy ai-session-*旧 conversation/messages，没有 provider threadId只能查看，或经用户确认引用历史后新建 native thread新 thread，不得伪装成 resume

对 Zeus 的结论

未来由 Zeus 创建的 native conversation 可以与 Codex 完全同构：持久化 provider threadId，停止后按需 thread/resume，然后在同一 thread 上启动新 turn。当前 4 个 legacy conversation 没有 threadId，只能保留为历史，或由用户显式确认引用后开启新的 native thread。
