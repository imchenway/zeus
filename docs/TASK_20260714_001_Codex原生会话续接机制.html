代码时序：Codex 依靠 threadId 与 rollout 原生续接

Codex 不靠常驻模型进程，也不靠把最近消息重新拼进 prompt。真正的连续会话由持久 threadId 标识，历史写入本地 rollout；同进程直接追加 turn，跨进程先恢复 thread，再追加 turn。

产品客户端Codex App / Zeus Renderer
持有并选择 conversation
客户端桥接层Zeus Manager / Coordinator
保存 providerThreadId
Codex app-serverJSON-RPC：thread/*、turn/*
维护当前加载状态
持久会话~/.codex/sessions/.../rollout-*.jsonl
保存 thread 历史
A · 首次创建并物化历史
1. 初始化连接initialize → initialized
2. 创建 threadthread/start
携带 cwd、model、sandbox
3. 返回身份返回稳定 thread.id；客户端必须持久化
4. 尚未必可恢复只有空 thread、没有真实 turn 时，可能还没有 rollout
5. 用户发消息输入属于同一个会话
6. 开始 turnturn/start(threadId, input)
7. 流式事件turn/started、item/*、turn/completed
8. 持久化 rolloutthread、turn、item 与上下文落盘，可供以后恢复
B · 同一 app-server 进程内继续
1. 选择历史 threadthread 当前为 loaded / idle
2. 直接追加新 turn无需再次 resume；继续调用 turn/start，仍使用原 threadId
3. 连续对话每轮 turnId 不同，但 threadId 不变
C · app-server 退出或 thread 已卸载后继续
1. 新进程初始化新的 generation，内存中没有旧 thread
2. 请求恢复thread/resume({threadId, cwd})
3. 加载历史app-server 按 threadId 找到 rollout，恢复模型可见状态
4. 返回同一 thread返回的 thread.id 仍是原值，并重新订阅事件
5. 客户端对账可用 thread/read(includeTurns) 校准本地 snapshot
6. 继续新一轮恢复完成后再次调用 turn/start；新 turn 追加到原 thread
7. 更新时间单独 resume 不更新 updatedAt；真正开始 turn 后才更新
D · 边界：续接、分叉与 legacy 不是一回事
原生续接thread/resume 返回同一 threadId；后续 turn 追加到原历史
原生分叉thread/fork 复制已存历史，但创建新的 threadId
Zeus legacy 不能原生续接旧记录只有 ai-session-* 或 null，没有 Codex threadId；不能伪装 thread/resume。保真路径只能是保留旧记录，并显式迁移上下文到新的 native thread。
能真正续接存在真实 provider threadId，且对应 rollout 可读取；恢复后继续使用同一 threadId。
能分叉但不是续接源对象本身是 Codex 持久 thread；thread/fork 创建新 threadId，并保留来源关系。
不能原生续接只有展示消息、旧 Runtime sessionId 或普通数据库行；缺少 provider thread 身份时不能声称 resume。
