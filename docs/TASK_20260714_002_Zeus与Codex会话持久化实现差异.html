Zeus 与 Codex 会话持久化：为什么不能只复制一个 JSONL

可以复制 Codex App 的实现边界，但“复制实现”不是让 Zeus 手写 rollout 文件，也不是在对话框里增加续接模式按钮；正确路径是显式导入一次，再由同一个 Rust ThreadStore 维护 rollout、metadata、状态索引和幂等账本。

为什么会话机制属于 Rust app-server？

1 会话不是 UI 文本列表

一个 thread 同时包含模型上下文、turn/item 事件、工具调用、审批、cwd、sandbox、模型配置与运行状态。

→
2 这些状态由 Agent Runtime 产生

真正知道某个 item 何时开始、完成、取消或需要恢复的是运行模型与工具循环的 Codex Core，而不是 Electron Renderer。

→
3 持久化必须与运行状态同源

Rust ThreadStore 在同一边界内追加 rollout、更新 metadata/state，并提供 start、resume、fork、archive 等一致语义。

已确认事实 OpenAI 将 app-server 定义为富客户端的认证、会话历史、审批与流式 Agent 事件接口；Codex App 通过协议使用它，而不是自己解释 rollout。
工程推断，不是 OpenAI 官方动机声明 选择 Rust 不是因为“JSONL 只有 Rust 能写”，而是 Codex Core 本身已经用 Rust 实现。把 ThreadStore 留在同一进程，可避免 Electron 与 Runtime 各维护一套状态机、协议类型和崩溃恢复逻辑。类型安全、并发与单二进制部署是附加收益，不是唯一原因。

“复制 Codex”需要承担哪些工程成本？
  • 随 Zeus 分发固定版本、按架构构建并签名的 Codex Rust app-server，同步 Apache 2.0 attribution。
  • 增加窄范围 zeus-legacy source provider，复用现有 externalAgentConfig/* 协议，不向 Zeus 暴露 ThreadStore 文件细节。
  • 为 Codex 版本升级建立 schema fingerprint、binary checksum、导入幂等、跨版本 resume 和回滚验证。
  • 失败时保留 legacy SQLite 记录；只有新 thread 已持久化且可 resume 后才绑定并归档来源。
