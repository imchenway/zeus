Zeus 会话记录、悬浮控件与权限模式故障链

现场不是一个“按钮难看”的孤立问题，而是展示投影、滚动锚点与权限状态三条链同时偏离 Codex App：空 provider item 被当正文、滚动按钮参与内容流、审批 UI 与真正执行策略互相矛盾。

4 个用户可见故障 根因已由源码 + SQLite + Codex App bundle 交叉确认 当前仅完成 PLAN，源码未修改

Before · 当前故障逻辑

截图中的真实链路
输入：provider item 与任务权限进入同一会话 SQLite 保存空 reasoning 壳、完整 commandExecution payload；任务 allow_code_changes=0。
无展示投影
根因 R1 · 所有 item 都生成可见 article 空 reasoning 仍显示“工作记录”；命令虽有 command/cwd/status/output，却退化成“工具调用 + 技术详情 + raw JSON”。E1 + E2
DOM 继续堆高
根因 R2 · 返回按钮位于滚动内容内部 session-return-latest 排在 transcript item 与 pending request 之间，并使用 position: sticky; bottom:118px，所以随内容流漂移。E3
用户被打断
根因 R3 · UI 审批与服务端策略不一致 thread/turn 固定 untrusted + user；只读任务先天拒绝 grant，含 shell/pipe 的请求也会被强制 decline，但 UI 仍展示“允许一次”。E4 + E5
产生矛盾反馈
结果：空白、漂移、低信息密度、伪可操作审批 用户无法判断工作进度；滚动位置被悬浮控件遮挡；点击“允许”也不能改变结果；设置页没有真正的权限控件。
删除错投影 · 新增模式源 · 保留真实事件

After · Codex App 对齐目标

同一状态源驱动 UI 与 RPC
输入：会话 permission profile 成为 native 权限事实源 composer 提供 Codex Agent Mode；新 thread 使用项目默认，既有 thread 显示当前 mode，变更作用于下一 turn。
映射一次
修法 F1 · 原样采用 Codex mode → RPC 映射 read-only、auto、full-access；granular / guardian 仅在 app-server capability/config requirements 允许时出现。F1
建立展示投影
修法 F2 · 真实事件不丢失，空壳不占行 空 reasoning 保留在持久化层但不渲染；有摘要的 reasoning 合并为一个轻量“工作进度”；命令显示动作、状态、耗时和按需输出，不默认暴露 raw JSON。
控件退出内容流
修法 F3 · 滚动控制属于 viewport overlay “返回最新消息”锚定 transcript viewport / composer 上沿，绝不作为消息 item 排版；只有离开底部阈值时出现，回到底部立即消失。
审批与策略一致
输出：只有真正可执行的审批，且结果不被静默改写 UI 只展示 app-server advertised decisions；用户选择经 schema/path 校验后原样发送。Full access 使用 dangerFullAccess + never，不再生成普通命令审批。
回滚点 / 安全边界 mode、投影、overlay 可分别回滚；Full access 必须显式选择、醒目风险提示、不得设为静默默认；不修改 provider JSONL/SQLite 原始 item。
