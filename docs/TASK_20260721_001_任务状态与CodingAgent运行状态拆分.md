# 任务状态与 Coding Agent 运行状态拆分

## 需求背景

任务列表原先把项目管理进度、Agent 执行状态和下一步动作混在同一列，同时额外展示“AI 执行”和“证据”，造成字段重复且语义不清。任务状态与
Coding Agent 会话运行状态必须拆成两套互不覆盖的事实。

本次需求：

1. 说明并简化任务编码中的序号展示。
2. “任务 / 意图”改为“任务”，单元格只显示标题，使用普通字重和较小字号。
3. 原“状态 / 下一步”改为“运行状态”。
4. 移除“AI 执行”和“证据”列。
5. 新增独立的“任务状态”列。

## 领域模型

### 任务编码与序号

`taskSequence` 是项目内递增序号，用于生成稳定、可读的 `taskCode`，例如序号 `1` 生成 `ZEU-000001`。两者在列表中同时展示属于重复信息，因此列表只显示
`taskCode`，但数据库继续保留 `taskSequence`，供编码生成和项目内排序等内部能力使用。

### 任务状态

任务状态表示项目管理阶段，由用户维护，不由 Coding Agent 会话自动改写：

| 值                     | 中文显示 | 含义                |
|-----------------------|------|-------------------|
| `todo`                | 待开始  | 已创建，尚未进入开发        |
| `in_development`      | 开发中  | 正在开发或修改           |
| `in_testing`          | 测试中  | 正在测试或验证           |
| `awaiting_acceptance` | 待验收  | 开发和测试已完成，等待验收     |
| `blocked`             | 已阻塞  | 因依赖、决策或环境问题暂时无法推进 |
| `completed`           | 已完成  | 项目管理意义上的工作已完成     |
| `cancelled`           | 已取消  | 任务不再继续            |

新任务默认 `todo`。任务状态可以通过任务列表的批量操作修改；修改任务状态不改变内部 Agent 执行状态，也不启动、暂停或终止会话。

### 运行状态

运行状态是只读派生值，来自任务关联的最新 Codex native 会话及其实时状态：

| 值                  | 中文显示   | 含义                    |
|--------------------|--------|-----------------------|
| `not_started`      | 未启动    | 任务还没有关联会话             |
| `connecting`       | 正在连接   | 会话正在连接或恢复现场           |
| `reconnecting`     | 正在重连   | 连接中断后正在重建             |
| `running`          | 运行中    | Coding Agent 正在处理当前轮次 |
| `waiting_user`     | 等待用户回复 | Agent 需要用户补充输入        |
| `waiting_approval` | 等待用户批准 | Agent 正在等待权限或动作批准     |
| `paused`           | 已暂停    | 会话队列已暂停               |
| `idle`             | 等待新指令  | 会话可继续，但当前没有运行中的轮次     |
| `failed`           | 运行失败   | 会话或当前轮次失败             |
| `legacy_readonly`  | 历史只读   | 历史会话只能查看，不能原生续接       |

运行状态不写入任务管理字段。实时会话状态优先；没有实时连接时使用会话持久化状态；没有会话时显示“未启动”。

### 旧数据迁移

新增 `tasks.management_status` 字段和项目、状态、更新时间组合索引。迁移只在首次执行时根据旧任务内部状态给出项目状态初值：

- `completed` → `completed`
- `cancelled` → `cancelled`
- `running`、`paused`、`waiting_confirmation`、`failed` → `in_development`
- 其他 → `todo`

迁移完成后，两套状态完全独立，不再自动互相同步。

## 列表设计

默认列顺序：

1. 任务编码
2. 任务
3. 任务状态
4. 运行状态
5. 上下文来源
6. 创建时间
7. 更新时间

“任务”列只显示标题，字号 `12px`、字重 `400`；不再显示描述摘要。任务编码列不再显示“序号 N”。“AI 执行”“证据”“状态 /
下一步”不再属于可配置列。

## 数据与接口

- 列表接口支持 `managementStatus` 筛选和 `managementStatus` 排序。
- 新增 `PATCH /api/tasks/:taskId/management-status`，仅接受任务状态枚举。
- 状态修改写入任务时间线、审计日志和实时 `task.updated` 事件。
- 导入导出包含 `management_status`，旧数据缺失时按迁移规则处理。
- Renderer 在任务页加载任务关联会话，确保运行状态不是因为未加载会话而误显示为“未启动”。

## 正式包现场回归修复

首次正式包验收发现，真实数据库中的旧自定义列偏好已处于“半迁移状态”：`visibleColumnKeys` 只剩任务编码、任务、来源、更新时间，但
`columnOrder` 末尾仍保留 `nextAction`、`aiExecution`、`signals`。Renderer 删除旧键后没有把新状态列补回，导致任务状态和运行状态未显示。

修复策略：

- Local Server 与 Renderer 同时识别旧列键，避免任一层再次返回或保存旧列模型。
- 只要可见列或顺序中仍存在旧键，就把新列迁移到“任务”之后，并补为可见列。
- 新版设置中已经不存在旧键时，继续尊重用户主动隐藏任务状态或运行状态的选择，不重复强制补回。
- 增加真实“半迁移状态”回归用例，覆盖正式包现场数据库中的列数组形态。

## 表格跨分辨率对齐契约

正式包截图进一步暴露出“运行状态”表头左对齐、数据右对齐的问题。根因是运行状态数据单元格仍继承旧任务列表 `.task-list-meta`
的右对齐规则，而表头只使用表格列类。

统一策略：

- 选择框列居中；任务编码、任务、任务状态、运行状态、上下文来源及其他文本列左对齐。
- 更新时间、创建时间等时间列右对齐，并使用等宽数字；表头和值共享同一个右侧锚点。
- 表头和数据行继续共享同一套 `--task-table-grid-template`，每个列键显式绑定 `start` 或 `end` 对齐类，不再借用旧列表元信息样式。
- 表头使用 `11px`、字重 `680`、更高文字对比度和轻量底部分隔线，提高辨识度但不恢复后台式重色块。
- 桌面宽度不足时保持统一最小轨道并使用横向滚动；`760px` 以下切换为“字段名 / 值”双列结构，不显示失去对应关系的裸值。

验收项：

- [x] “运行状态”表头和值左边缘一致。
- [x] 时间列表头和值右边缘一致。
- [x] 表头在字号、字重、颜色和分隔关系上明显区别于数据行。
- [x] 窄屏行包含真实字段名，不依赖已隐藏的桌面表头理解数据。

正式包验证覆盖默认宽窗口与约 `773px` 窄窗口：运行状态、任务状态等文本列保持左侧锚点，更新时间保持右侧锚点，列头在浅色表头底面和分隔线中可明确识别。尝试通过本地
HTTP 渲染入口补测 `700px` 断点时，页面因缺少 Electron preload 桥接按设计拒绝 hydration；未把该空白页计为视觉通过。`760px`
以下的字段名/值结构由静态渲染 HTML 与 CSS 断点回归测试覆盖。

## 运行状态对齐与创建时间补充

后续正式包截图显示，运行状态虽已绑定左对齐策略，单元格仍残留旧版状态组合布局的 `inline-flex`、基线对齐和间距
class，导致其视觉起点与任务状态不一致。此次直接移除运行状态的遗留布局 class，使任务状态、运行状态的表头和值都只受统一表格网格和
`start` 对齐策略控制。

默认任务列新增“创建时间”，位于“更新时间”之前。Renderer 与 Local Server
同步升级默认列模型；仅当可见列和列顺序都完整匹配上一正式包的默认配置时，自动迁移为新默认配置。这样既能让现有默认视图直接出现创建时间，也不会覆盖用户主动调整过的自定义列视图。用户在新版中主动隐藏创建时间后，列顺序仍为新版结构，不会被重复迁移。

新增回归覆盖：

- 运行状态不再包含旧版 `task-table-status-cell` 和 `task-table-runtime-cell`。
- 默认列顺序为“上下文来源、创建时间、更新时间”。
- 上一正式包的默认列偏好会自动迁移。
- 非默认的自定义可见列和列顺序继续保留。

当前正式包验证：`pnpm package:mac` 成功生成 App、DMG 和 ZIP，`codesign --verify --deep --strict dist/mac-arm64/Zeus.app`
通过。重新启动正式 App 后，真实任务表格显示“任务编码、任务、任务状态、运行状态、上下文来源、创建时间、更新时间”；任务状态和运行状态的表头与行值使用相同左侧锚点，创建时间和更新时间使用相同右侧锚点。

## 正式截图二次返修：共同盒模型

上一轮只验证了表头和数据单元格使用同一网格变量及对齐 class，却没有测量真实渲染坐标，因此错误地把代码结构一致当成视觉对齐完成。正式
`tc-app-core` 截图再次证明验收结论不成立。

通过 Electron CDP 读取真实 DOM 几何位置后确认：数据行设置 `inline-size: 100%`，同时仍使用默认 `content-box` 并带左右各
`12px` padding，导致数据行实际总宽度比表头多 `24px`。两者虽然共享相同的 `fr` 网格定义，但计算基准宽度不同，偏差会从左向右累积：任务状态偏
`12px`、运行状态偏约 `15.63px`、上下文来源偏约 `20.05px`。

最终修复：

- 表头和数据行统一使用 `box-sizing: border-box` 与 `inline-size: 100%`，保证网格的总宽度计算完全相同。
- 创建时间和更新时间默认轨道从 `112px` 扩为 `128px`，完整容纳 19 位时间文本，避免左侧年份被裁掉。
- 回归测试同时约束共享网格、共同盒模型及时间列最小宽度。

重打正式包后的真实几何验收：表头宽度和首行宽度均为 `987px`；任务状态、运行状态、上下文来源的文字左锚点偏差均为 `0px`
；创建时间、更新时间的文字右锚点偏差均为 `0px`；两列时间文本的 `clipped` 检查均为 `false`。

## 时间列头视觉重心调整

“创建时间”和“更新时间”列头改为列内居中，时间数据继续右对齐。短列头用于明确两个时间分区，19 位时间值则继续共享右侧数字扫描基线，避免整列数据参差。

正式包几何验收：

- 两个时间列头的 `justify-items` 和 `text-align` 均为 `center`。
- 两个列头文字中心相对单元格中心的偏差约为 `-0.004px`，可视为完全居中。
- 数据行仍为 `end` 对齐，时间文字右边缘与单元格右边缘偏差均为 `0px`。
- 创建时间与更新时间均未发生裁切。
